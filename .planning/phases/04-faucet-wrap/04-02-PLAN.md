<objective>
Ship the Wrap slice (WRP-01, WRP-02): a `/wrap` screen where a judge enters an ERC-20
amount, sees an **onchain-accurate preview** (rate() + per-side decimals, rounded down,
below-one-unit warning), runs a single approve→wrap flow via `useShield`, and watches a
4-stage indicator progress to done — then reveals the wrapped balance with the Phase-3
decrypt to prove **decrypt == preview** (the top judging axis).

`useShield` (installed 3.0.0) orchestrates ERC-20 approval + wrap in ONE mutation; its
`onApprovalSubmitted` / `onShieldSubmitted` callbacks drive the indicator. Do NOT
hand-roll approve+wrap. The preview is pure `bigint` math — the only genuinely new,
unit-testable app logic (`rate = 10^(underlyingDec − wrapperDec)`; wrappers are 6-dp).

Purpose: The second half of the demo path, made visible by Phase 3. Independent of the
Faucet slice (04-01) — no shared files — so both run in parallel (Wave 1).
Output: `lib/previewWrap.ts` (+test), `lib/wrapErrors.ts` (+test), `hooks/useWrap.ts`,
`components/wrap/*`, `app/wrap/page.tsx`, and the wired PairCard `Wrap →` CTA.
</objective>

  <behavior>
    previewWrap(underlyingRaw: bigint, rate: bigint, wrapperDecimals: number)
      -> { confRaw: bigint, consumedRaw: bigint, refundRaw: bigint, belowOneUnit: boolean }:
      - rate=1n (cUSDC, 6-dp underlying): underlyingRaw=1_000_000n -> confRaw=1_000_000n, consumedRaw=1_000_000n, refundRaw=0n, belowOneUnit=false ("1 whole underlying -> 1.0 confidential" @ 6-dp)
      - rate=1_000_000_000_000n (cWETH, 18-dp underlying, wrapper 6-dp): underlyingRaw=1_000_000_000_000_000_000n (1e18) -> confRaw=1_000_000n (=1.0 cWETH), consumedRaw=1e18, refundRaw=0n ("1 whole underlying -> 1.0 confidential" @ 18-dp — this is why hardcoding 18 is fatal)
      - below one unit: rate=1e12, underlyingRaw=100_000_000_000n (1e11 = 0.0000001 WETH) -> confRaw=0n, belowOneUnit=true
