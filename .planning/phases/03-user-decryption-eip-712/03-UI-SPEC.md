# Phase 3 — UI Design Contract (User-Decryption / EIP-712 · Decrypt blur→reveal)

> Visual and interaction contract for the confidential-balance decryption screen. **Transcribed from the locked design source of truth** `.planning/design/Confidential-Wrapper-Registry.dc.html` ("The Cellar Registry" — the `isDecrypt` / `route:'decrypt'` screen, lines 242–279 + derived state 636–776). Every token, string, animation, and state below is lifted verbatim from that file; the only additions are the reconciliation of the mock's registry-only limitation with the real DEC-01…DEC-04 requirement (decrypt ANY ERC-7984, single reusable EIP-712 permit, graceful no-ACL-access). Do NOT invent a new aesthetic. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** DEC-01…DEC-04 — a connected wallet decrypts its confidential (ERC-7984) balance via a correct EIP-712 user-decryption flow, for registry cTokenMocks AND any pasted ERC-7984 address, with a blur→reveal treatment. Built early because it is the only way to *see* the result of wrap (Phase 4) and unwrap (Phase 5) — build it to be reusable by those phases. First phase to import `@zama-fhe/sdk` + `@zama-fhe/react-sdk` (pinned EXACT **3.0.0** — verify hook names via context7; the CLAUDE.md hook map lists 3.2.0 names).

**Theme inheritance:** All CSS vars, fonts (Gelasio + JetBrains Mono, self-hosted via `next/font`), the two themes (`parchment`/`cellar`), the hard "letterpress" shadow, and the focus-visible outline are inherited verbatim from `02-UI-SPEC.md`. This spec only declares what is NEW or specific to the decrypt screen.

## Interaction Contract

### Decrypt state machine (`dStage`, from source lines 639–776)

| Stage | State chip | CTA label | CTA enabled? | Notes |
|-------|-----------|-----------|--------------|-------|
| (not connected) | `◈ ENCRYPTED CIPHERTEXT` (blue) | `Connect wallet to decrypt` | no | Gated by connect + `ChainGuard` |
| `idle` (no permit) | `◈ ENCRYPTED CIPHERTEXT` (blue) | `Sign & Decrypt (EIP-712)` | yes (valid token entered) | Blurred ciphertext shown |
| `idle` (permit active) | `◈ ENCRYPTED CIPHERTEXT` (blue) | `Decrypt` | yes | Skips signing step (DEC-03 cached permit) |
| `signing` | `✎ AWAITING EIP-712 SIGNATURE…` (red) | `Check your wallet — sign…` | no (busy) | Wallet prompt open |
| `decrypting` | `◌ GATEWAY DECRYPTING…` (red) | `Decrypting via gateway…` | no (busy) | Relayer in-flight |
| `revealed` | `✓ DECRYPTED BALANCE` (green) | `Decrypt another` | yes | Cleartext + `revealNum` animation |
| `error` | `✕ {REASON}` (red) | reverts to `Sign & Decrypt (EIP-712)` | per reason | See error rows in Copywriting |

| Interaction | Behavior |
|-------------|----------|
| Enter / paste address | Trim + `isAddress` validate on change; invalid → `✕ NOT A VALID ADDRESS`, CTA disabled. Resets stage to `idle` |
| Pick a registry chip | Sets address to that cToken's confidential address, resets to `idle`, highlights chip (blue border/fill) |
| Click Sign & Decrypt (no permit) | `idle → signing` (EIP-712 wallet prompt) `→ decrypting` (gateway) `→ revealed`. On reject at `signing` → back to `idle` + `Signature declined.` toast. On gateway failure at `decrypting` → `✕ DECRYPTION FAILED` + retry |
| Click Decrypt (permit active) | Skips `signing`; `idle → decrypting → revealed` (no repeat wallet prompt within the permit window — DEC-03) |
| Click Decrypt another | Clears the address + returns to `idle` (source line 772) |
| No ACL access (DEC-04) | After the decrypt attempt resolves to "no access", show `✕ NO DECRYPTION ACCESS` + guidance — never a hanging spinner |
| Blur → reveal transition | Idle value has `filter:blur(9px)` + `inkPulse 2.6s` (breathing ciphertext); on reveal the cleartext number animates `revealNum 0.9s cubic-bezier(0.2,0.8,0.2,1)` (blur(12px)/opacity 0.3 → blur(0)/opacity 1). Reduced-motion: skip both animations, show final state directly (blur→reveal is functional, so present the revealed value without motion) |
| Per-card decrypt (`PairCardDecrypt`) | Same state machine + same blur→reveal treatment inline on the Phase-2 card, connected+Sepolia only; shares `useUserDecrypt` so a granted permit reveals across cards without re-signing |
| Theme toggle | Existing Phase-1 mechanism flips `data-theme`; all decrypt tokens follow via CSS vars |

## Layout & Responsive

- Page shell: sticky `Header` (Phase 1, with new `Decrypt` nav route) → Phase-1 `ChainGuard` wrong-network banner → `<main>` (Phase-2 shell, `max-width:1180px`).
- `DecryptPanel`: `<section max-width:880px; margin:0 auto>`, centered header `margin-bottom:30px`.
- Panel grid: `minmax(200px, 260px) 1fr` — the illustration column collapses under the content column on narrow viewports; below ~560px stack to a single column (illustration above, or hide illustration to keep the form usable — illustration is decorative).
- Right column: `padding:26px`, flex column `gap:18px`; quick-pick chips wrap (`flex-wrap:wrap`).
- Focus-visible: `2px solid --red` outline, `offset:2px` on the input, chips, and CTA (inherited global).
- Reduced motion: `inkPulse` (idle) and `revealNum` (reveal) are the only animations — keep `inkPulse` subtle; respect `prefers-reduced-motion` by rendering final states without motion. No essential information is conveyed by animation alone.
